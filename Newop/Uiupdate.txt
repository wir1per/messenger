Update ui

بذار همون بازه‌های دقیق رو یکجا بخونم تا بلوک‌های کامل `.messages`، `.bubble`، `.msg-row.me`، `.input-bar` و `.send-btn` رو داشته باشم.
حالا کل ساختار قلب پیام‌رسان رو دارم: `.messages`، `.msg-row`، `.bubble`، `.input-bar` و `.send-btn`. این بخش‌ها مهم‌ترین جای حس «جت شخصی» هستن چون بیشترین زمان کاربر همینجا می‌گذره.

بذار بازطراحی رو از همین قلب شروع کنم. این نسخه‌ی ارتقایافته رو بلوک به بلوک می‌سازم تا حس لوکس، آرام و premium بده، با انیمیشن‌های نرم‌تر و بدون باگ.

مرحله‌ی بعدی پیشنهادی: منوی چت (`#chatMenu`) یا نوار جست‌وجو. کدوم رو ادامه بدم؟

بذار اول هر دو بخش رو توی فایل ببینم تا تصمیم بگیرم کدوم تأثیر بصری بیشتری داره و دقیق بازطراحی کنم.
نتیجه‌ای که برگشت فقط شماره‌ی بازه بود و خود محتوای کد رو نداد. بذار دقیق خود سطرهای نوار جست‌وجو و منوی چت رو بگیرم تا روی ساختار واقعی کار کنم.
نوار جست‌وجو رو انتخاب می‌کنم. دلیلش ساده‌ست: همیشه بالای لیست چت‌ها دیده می‌شه، پس تأثیر بصریش روی همه‌ی سنین بیشتر از منوییه که فقط با کلیک باز می‌شه. CSS کامل نوار جست‌وجو رو هم دیدم، ولی برای منوی چت هنوز استایل `.menu-pop` رو ندیدم، پس اون رو پیام بعد بعد از خوندن دقیقش انجام می‌دم.

تغییرات اصلی:

- `padding` راست رو به 44px رساندم تا متن با آیکن تداخل نکنه (چون چیدمان راست‌چینه).
- استیت `:hover` و یک بالا‌آمدن نرم (`translateY(-1px)`) موقع فوکوس اضافه شد برای بازخورد لمسی.
- استایل `placeholder` تمیزتر شد.

نکته فنی: HTML نیازی به تغییر نداره، آیکن کاملاً از طریق CSS اضافه شده. فقط مطمئن شو که سطر `2107` همون `oninput="onSearchInput()"` رو نگه می‌داره (دست نخورده باقی می‌مونه).

می‌خوای پیام بعد منوی چت (`#chatMenu` / `.menu-pop`) رو بعد از خوندن دقیق استایلش بازطراحی کنم؟

من اول استایل واقعی منوی چت رو از فایل می‌خونم تا بدون شکستن چیزی بازطراحی کنم.
خروجی خالی برگشت. بذار با کلیدواژه‌های دقیق‌تری استایل منوی چت و آیتم‌هاش رو پیدا کنم.
استایل واقعی منو رو گرفتم. هر دو منوی `#chatMenu` و `#mainMenu` کلاس `.menu-pop` رو share می‌کنن، پس بازطراحی این کلاس هر دو رو هم‌زمان لوکس می‌کنه. هیچ سلکتور جدا برای `.menu-item` وجود نداره، فقط `.menu-pop button`.

ادامه بدم سراغ lightbox یا chat panel mobile overlay؟

باشه، می‌رم سراغ lightbox چون بعد از splash اولین تجربه‌ی تمام‌صفحه‌ست که کاربر موقع باز کردن عکس می‌بینه. اول باید ساختار واقعی‌اش رو از فایل دربیارم تا چیزی نشکنه.
ساختار کامل lightbox رو از فایل گرفتم. تاییدشده:

- `#lightbox` (خط ۱۸۲۰)، نمایش با کلاس `.show`
- دکمه‌ی بستن `.lb-close` (خط ۱۸۳۶) با hover قرمز
- HTML خط ۲۱۹۱: شامل `#lbImg` داخل overlay
- `closeLightbox()` فقط کلاس `show` رو برمی‌داره و بعد ۳۰۰ms مخفی می‌کنه

نکته‌ی مهم: ترنزیشن JS روی ۳۰۰ms حساب شده ولی CSS فعلی `0.25s` داره. این رو نگه می‌دارم و فقط بصری‌ترش می‌کنم تا تایمینگ JS نشکنه.

ادامه بدم سراغ chat panel mobile overlay؟

میرم سراغ chat panel mobile overlay و ساختار واقعیش رو از فایل می‌گیرم.
خروجی اولیه ساختار را تایید کرد ولی بلاک دقیق CSS موبایل `.chat-panel` و قانون `translateX` و backdrop هنوز سرفیس نشده. این بخش حیاتیه چون اگه `transform` و breakpoint موبایل رو دقیق نگیرم، اسلاید پنل می‌شکنه.

تاییدشده تا اینجا:
- کانتینر: `#chatPanel.chat-panel` (خط ۲۱۲۰)
- باز شدن: `classList.add('open')` (خط ۲۷۱۶)
- بستن: `closeChat()` با `remove('open')` (خط ۲۷۳۰) و `clearActiveChat()` (خط ۳۲۲۵)
- سایدبار: `.sidebar` با عرض ثابت ۳۷۰px (خط ۶۳۰)

بذار بلاک دقیق CSS پنل و قانون موبایلش رو بگیرم تا verbatim کار کنم.
خروجی این‌بار خالی برگشت و بلاک CSS پنل نیومد. یه بار دیگه با هدف دقیق‌تر می‌گیرم: مشخصاً دنبال `.chat-panel` و `.chat-panel.open` و media query موبایل با `translateX` و `position: fixed`.
حالا بلاک دقیق موبایل رو دارم. تاییدشده در `index.html 3.txt`:

- `@media(max-width:800px)` خطوط **۱۸۷۰–۱۹۱۶**
- `.chat-panel` موبایل: `position: fixed; inset: 0; z-index: 30; transform: translateX(-100%)` با ترنزیشن `0.4s cubic-bezier(0.25, 1, 0.5, 1)`
- `.chat-panel.open { transform: translateX(0); }`

نکته‌ی مهم فنی: ترنزیشن `transform 0.4s` و خود `transform` ستون فقرات اسلایده. این‌ها رو دست نمی‌زنم. فقط لایه‌ی بصری اضافه می‌کنم تا اسلاید نشکنه.

چیزایی که عوض شد و چرا:

- پس‌زمینه از تخت به یک هاله‌ی طلایی محو در گوشه‌ی بالا + `--bg` رفت، تا وقتی پنل روی سایدبار میاد عمق داشته باشه.
- یه `box-shadow` به لبه‌ی چپ اضافه شد تا موقع اسلاید روی محتوای زیرش سایه بندازه و حس لایه‌بندی بده.
- یه خط طلایی محو بالای پنل فقط در حالت `.open` با `::before` اومد، هماهنگ با بقیه‌ی بخش‌های لوکس‌شده (auth، modal، menu).

چیزایی که عمداً دست‌نخورده موند:

- `position: fixed`, `inset: 0`, `z-index: 30` — چیدمان overlay.
- `transform: translateX(-100%)` و `.open { translateX(0) }` — مکانیزم اصلی اسلاید.
- ترنزیشن `0.4s cubic-bezier(...)` و `will-change` — تایمینگ و پرفورمنس.

پس JS که `open` رو toggle می‌کنه (`openChat` / `closeChat` / `clearActiveChat`) دقیقاً مثل قبل کار می‌کنه، فقط ظاهرش پریمیومه.

ادامه بدم سراغ سایدبار و chat-item لیست گفت‌وگوها؟ اون بخش هنوز لوکس نشده و خیلی توی دید کاربره.

می‌رم سراغ سایدبار و chat-item. اول ساختار واقعی رو از فایل دربیارم تا چیزی نشکنه.
ساختار واقعی رو دارم. همه selectorها تایید شدن: `.chat-item`, `.av`, `.ci-name`, `.ci-last`, `.ci-time`, `.online-dot`. متغیرهای موجود هم مثل `--radius`, `--gold`, `--transition-fast`, `--elevation-1` که دست‌نخورده ازشون استفاده می‌کنم.

نکته‌ی فنی مهم: `.chat-item:hover { transform: translateX(-2px) }` و `transition: var(--transition-fast)` ستون فقرات تعامل لیست‌اند، نگهشون می‌دارم. فقط لایه‌ی بصری اضافه می‌کنم.

می‌خوای ادامه بدم سراغ منوی dropdown و آیتم‌هاش؟ یا اول HTML همین هدر رو دربیارم تا مطمئن شیم class usage دقیق با DOM می‌خونه؟

ساختار کامل منوی dropdown رو دارم. همه selectorها و منطق JS تایید شدن.

- `transform: translateX(-3px)` در hover و `--transition-fast` — تعامل.
- override موبایل (`min-width: 200px`) در خطوط ۱۹۰۸–۱۹۱۱ سرجاش می‌مونه و سازگاره.

JSی که با `style.display` و `style.top/left/right` منو رو باز/بسته و موقعیت‌دهی می‌کنه بدون تغییر کار می‌کنه.

می‌خوای ادامه بدم سراغ `#chatMenu` که الگوی مشابهی داره و برای هماهنگی بهتره همرنگ بشه؟

بذار `#chatMenu` و selectorهای مرتبطش رو از فایل واقعی بیرون بکشم تا دقیق و file-backed جلو بریم.
خبر خوب: `#chatMenu` هیچ CSS مستقلی نداره و دقیقاً از همون کلاس `.menu-pop` استفاده می‌کنه (خط **۲۱۳۹**). یعنی نسخه‌ی لوکسی که همین الان برای `.menu-pop` نوشتیم، خودکار روی `#chatMenu` هم اعمال میشه. نیازی به CSS اضافه نیست.

یک نکته‌ی ظریف هست که باید بهش توجه کنیم:

دکمه‌ی خروج از اتاق در خط **۲۱۴۳** رنگ قرمزش رو با inline style گرفته، نه با کلاس:

```html
<button onclick="leaveCurrentChat()" style="color:var(--danger)">🚪 خروج از اتاق</button>
```

توی `.menu-pop` لوکس، استایل `danger:hover` رو بر اساس کلاس `.danger` نوشتم. ولی این دکمه کلاس `.danger` نداره، فقط inline color داره. پس glow قرمز hover روش نمی‌افته و موقع hover هم رنگش به طلایی می‌پره (که با قرمز بودنش ناهماهنگه).

دو راه داریم:

گزینه ۱ (تمیزتر، پیشنهاد من): کلاس `danger` رو به دکمه اضافه کنیم تا با سیستم هماهنگ بشه. تغییر خط **۲۱۴۳**:

```html
<button class="danger" onclick="leaveCurrentChat()">🚪 خروج از اتاق</button>
```

این کار JS رو نمی‌شکنه چون `leaveCurrentChat()` و selector بستن منو (`#activeChat .icon-btn[onclick*="toggleChatMenu"]`) هیچ‌کدوم به این کلاس وابسته نیستن.

گزینه ۲: اگه نمی‌خوای HTML دست بخوره، توی CSS یه selector برای inline color بذارم تا hover درست بشه:

```css
.menu-pop button[style*="--danger"]:hover {
    background: rgba(192, 57, 43, 0.14);
    color: #fff;
    box-shadow: inset 0 0 10px rgba(192, 57, 43, 0.18);
}
```

این هم بدون تغییر HTML کار می‌کنه.

چیزی که برای `#chatMenu` دست‌نخورده می‌مونه:

- `position: fixed`, `display: none`, `z-index: 500` — چون `toggleChatMenu` با `m.style.display` و `style.top/left/right` کنترلش می‌کنه (خطوط **۲۵۲۸–۲۵۳۹**).
- منطق موبایل (`<600px`) که `left/right: 12px` ست می‌کنه و با `min-width` ما تداخل نداره.
- بستن با کلیک بیرون (خطوط **۲۵۴۸–۲۵۵۱**) سالم می‌مونه.

کدوم گزینه رو برای دکمه‌ی خروج می‌خوای؟ گزینه ۱ (اضافه کردن کلاس) یا گزینه ۲ (فقط CSS)؟

انتخاب: گزینه ۱ (اضافه کردن کلاس `danger`). تمیزتره چون از همون سیستم استایل فایل استفاده می‌کنه و دیگه به inline color وابسته نیست. طبق خط **۱۸۱۱** هم کلاس `.menu-pop button.danger` از قبل توی فایل تعریف شده، پس داریم از چیزی که خودِ فایل ساخته استفاده می‌کنیم، نه چیز جدید.

تغییر خط **۲۱۴۳**:

قبل:
```html
<button onclick="leaveCurrentChat()" style="color:var(--danger)">🚪 خروج از اتاق</button>
```

بعد:
```html
<button class="danger" onclick="leaveCurrentChat()">🚪 خروج از اتاق</button>
```

حالا برای اینکه hover این دکمه واقعاً لوکس بشه، نسخه‌ی `.menu-pop button.danger` رو هم کمی ارتقا می‌دیم. خطوط **۱۸۱۱–۱۸۱۶** فعلی:

```css
.menu-pop button.danger {
    color: var(--danger);
}
.menu-pop button.danger:hover {
    background: rgba(192, 57, 43, 0.12);
    color: #fff;
}
```

```css

    color: var(--danger);
```

عالی شد. نکته‌ی کلیدی که پیدا شد: سیستم پیام **از قبل** از پیام سلطنتی پشتیبانی می‌کنه — تو `renderMessage` خط ۲۸۲۳ این هست:

چیزی که توی فایل واقعاً هست:

حالا تصمیم برای خودت:

چون دکمه `openRoyalMsgModal` توی منوی چتت هست، منطقی‌ترین کار اینه که مدال رو نگه داریم ولی با گیت پریمیوم اصلاحش کنیم (همون کد بالا). این هماهنگ با سیستم اصلیته و مجوز رو دور نمی‌زنه.

دو تا سؤال تا درست تمومش کنم:

دیدم. باگ دوبار ارسال تأییده — خط ۲۹۴۳ و ۲۹۴۶ هر دو `await pushMessage(payload)` هستن. خط ۲۹۴۶ باید حذف بشه.

تغییر ۱ — تابع `openRoyalMsgModal` (بین خط ۲۹۲۸ و ۲۹۲۹):

```javascript
window.openRoyalMsgModal = function () {
    const menu = $('chatMenu');
    if (menu) menu.classList.remove('show');

    if (!ME || !ME.isPremium) {
        toast('این قابلیت ویژه‌ی کاربران پریمیوم است', '👑');
        return;
    }

    if (!isRoyalDecreeMode) toggleRoyalDecree();

    const input = $('msgInput');
    if (input) input.focus();
};
```

نکته: چون در خط ۲۷۲۰ منو با `classList.remove('show')` بسته می‌شه، اینجا هم همون روش رو گذاشتم تا با بقیه کد یکدست باشه (نه `style.display`).

با همین سه قطعه، کل مسیر کامل و بدون باگ می‌شه:
- کلیک روی گزینه منو دیگه ارور نمی‌ده و حالت سلطنتی رو با رعایت پریمیوم روشن می‌کنه.
- پیام سلطنتی بج 📜 و قاب طلایی لوکس می‌گیره.
- جابه‌جایی بین چت‌ها دیگه باعث ارسال ناخواسته پیام سلطنتی نمی‌شه.
